            Why the binary search rule matters
          

          

            

              
Search step
In the optimized method, binary search finds the first tail that is greater than or equal to the current value. That exact pile must be updated to keep tails minimal.

              
Why it matters
If you search for the wrong pile, or append equal values instead of replacing them, the structure stops representing strictly increasing subsequences correctly.

          Overview
          

            
The Longest Increasing Subsequence problem asks for the longest subsequence whose values are strictly increasing. A subsequence keeps the original order, but it does not need to be contiguous.

            
This tool shows two valid solutions. The dynamic programming version computes the best answer ending at every index. The optimized version uses patience sorting ideas and binary search to keep one minimal tail for every possible subsequence length.

            
Use the DP mode to understand the recurrence clearly, then switch to the binary-search mode to see how the same problem can be solved faster.

          

        


        

          Worked Example
          

            
Example array: [10, 9, 2, 5, 3, 7, 101, 18]

            
DP eventually builds values like [1, 1, 1, 2, 2, 3, 4, 4]. That means the best subsequence ending at 101 has length 4.

            
One valid LIS is [2, 3, 7, 18]. Another valid answer is [2, 5, 7, 101]. The tool reconstructs one of the valid optimal subsequences using parent pointers.

          

        


        

          

            
Key Concepts + Common Mistakes

            For CLO-3
          

          

            

              
Dynamic ProgrammingOverlapping subproblems appear because each LIS[i] depends on earlier LIS[j] values. The DP table visualizes those dependencies directly.

              
Optimized SearchThe tails array does not always store an actual LIS. It stores the best possible ending value for each length, which is enough to compute the final answer.

              
Common mistakeConfusing subsequence with substring/subarray. LIS does not need adjacent elements; it only requires the original order to stay the same.

              
Wrong paradigm exampleA naive greedy strategy like “always take the next bigger number immediately” can fail. For [3, 4, 1, 2, 8, 5, 6], that greedy idea can get stuck with [3, 4, 8], while the optimal LIS is [1, 2, 5, 6].

            

          

        

      


      

        

          
Complexity + Paradigm Summary

          Best / average / worst case
        

        

          

            
DP paradigm
Dynamic Programming. Time: O(n²) in best, average, and worst case. Space: O(n).

            
Optimized paradigm
Still solves LIS exactly, but speeds up the transition step with binary search. Time: O(n log n) in best, average, and worst case. Space: O(n).

            
Worst case preset
Use the Worst Case button to load a decreasing array. For DP, every j is still checked for every i, so the number of comparisons is maximized.

            
Alternative approach is worse
A brute-force check of all subsequences is exponential, so both DP and the binary-search method are dramatically better.
